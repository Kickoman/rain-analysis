# GLOSSARY.md — Machine Learning Metrics

Определения метрик качества моделей машинного обучения, используемых в этом проекте.

---

## Confusion Matrix (Матрица ошибок)

Таблица, показывающая соотношение правильных и неправильных предсказаний модели по сравнению с реальностью.

|                    | **Predicted: Rain** | **Predicted: No Rain** |
|--------------------|:-------------------:|:----------------------:|
| **Actual: Rain**   | TP (True Positive)  | FN (False Negative)    |
| **Actual: No Rain**| FP (False Positive) | TN (True Negative)     |

### В контексте предсказания дождя:

- **TP (True Positive)** — модель предсказала дождь, и он действительно пошёл ✅
- **TN (True Negative)** — модель предсказала отсутствие дождя, и его не было ✅
- **FP (False Positive)** — модель предсказала дождь, но его не было ❌ (ложная тревога)
- **FN (False Negative)** — модель не предсказала дождь, но он пошёл ❌ (пропущенный дождь)

---

## Precision (Точность)

**Формула:** `Precision = TP / (TP + FP)`

**Значение:** Какая доля предсказаний дождя оказалась правильной.

**Интерпретация:**
- **Высокая precision (≥0.7)**: модель редко даёт ложные тревоги — когда она говорит "будет дождь", скорее всего так и будет.
- **Низкая precision (<0.5)**: много ложных тревог — модель часто предсказывает дождь, когда его нет.

**Пример:**  
Модель 100 раз предсказала дождь. Из них 70 раз дождь действительно пошёл (TP=70), 30 раз его не было (FP=30).  
`Precision = 70/(70+30) = 0.70` (70%)

---

## Recall (Полнота, Sensitivity)

**Формула:** `Recall = TP / (TP + FN)`

**Значение:** Какую долю реальных дождей модель смогла предсказать.

**Интерпретация:**
- **Высокий recall (≥0.7)**: модель находит большинство дождей — редко пропускает реальный дождь.
- **Низкий recall (<0.5)**: модель пропускает много дождей — часто не предупреждает, когда дождь начинается.

**Пример:**  
Было 100 реальных дождей. Модель предсказала 60 из них (TP=60), остальные 40 пропустила (FN=40).  
`Recall = 60/(60+40) = 0.60` (60%)

---

## F1 Score (F1-мера)

**Формула:** `F1 = 2 × (Precision × Recall) / (Precision + Recall)`

**Значение:** Гармоническое среднее между precision и recall. Сбалансированная метрика.

**Интерпретация:**
- **F1 ≥ 0.7**: отличное качество — модель одновременно точная и полная
- **F1 = 0.4–0.6**: средняя модель — есть компромисс между ложными тревогами и пропущенными дождями
- **F1 < 0.4**: слабая модель — либо много ложных тревог, либо много пропусков (или и то, и другое)

**Применение:**  
Используется, когда precision и recall одинаково важны. Если модель хороша только по одной метрике (например, высокий recall, но низкий precision), F1 будет низким.

**Пример:**  
Precision = 0.70, Recall = 0.60  
`F1 = 2 × (0.70 × 0.60) / (0.70 + 0.60) = 0.65`

---

## F-beta Score (F-β мера)

**Формула:** `F-beta = (1 + β²) × (Precision × Recall) / (β² × Precision + Recall)`

**Значение:** Обобщение F1, позволяющее придать больший вес либо precision, либо recall.

**Параметр β:**
- **β = 1**: равный вес precision и recall → **F1 Score**
- **β < 1** (например, 0.5): больший вес **precision** — штрафуем ложные тревоги сильнее
- **β > 1** (например, 2): больший вес **recall** — штрафуем пропущенные дожди сильнее

### F2 Score (F-2 мера)

**Формула:** `F2 = 5 × (Precision × Recall) / (4 × Precision + Recall)`

**Особенность:** Придаёт recall **в 4 раза больший вес**, чем precision.

**Применение в нашем проекте:**  
Для задачи предсказания дождя **пропустить реальный дождь хуже**, чем дать ложную тревогу:
- Пропущенный дождь (FN) → человек остался без зонта/одежды ❌❌
- Ложная тревога (FP) → человек взял зонт, но не понадобился ❌

Поэтому **F2 предпочтительнее F1** для оценки моделей предсказания дождя.

**Пример:**  
Precision = 0.60, Recall = 0.80  
`F2 = 5 × (0.60 × 0.80) / (4 × 0.60 + 0.80) = 0.74`

Несмотря на невысокий precision (0.60), F2 остаётся относительно высоким (0.74) благодаря отличному recall (0.80).

---

## Specificity (Специфичность)

**Формула:** `Specificity = TN / (TN + FP)`

**Значение:** Какую долю реальных "не-дождей" модель правильно определила.

**Интерпретация:**
- **Высокая specificity**: модель редко даёт ложные тревоги когда дождя нет.
- **Низкая specificity**: модель часто кричит "дождь!" без повода.

**Применение:**  
Менее важна в задаче предсказания дождя по сравнению с recall, но полезна для оценки "спокойствия" модели.

---

## Threshold (Порог срабатывания)

**Значение:** Минимальное значение выходной вероятности модели, при котором модель считает, что будет дождь.

**Пример:**  
Модель выдаёт вероятность дождя 0.65 (65%). Если threshold = 0.50, модель предсказывает дождь. Если threshold = 0.70, модель предсказывает отсутствие дождя.

**Компромисс:**
- **Низкий threshold (0.3–0.4)**: больше TP, но и больше FP → **высокий recall, низкий precision**
- **Высокий threshold (0.6–0.7)**: меньше FP, но и меньше TP → **высокий precision, низкий recall**

**Применение:**  
Подбирается так, чтобы максимизировать нужную метрику (F1, F2) на тестовых данных.

---

## Support

**Значение:** Количество реальных случаев в данных для данного класса.

**Пример:**  
Support для класса "Rain" = 150 означает, что в тестовых данных было 150 часов с дождём.

**Применение:**  
Помогает понять, насколько репрезентативны данные. Низкий support (например, 10 дождей) делает метрики менее надёжными.

---

## Примечания

### Trade-off между Precision и Recall

Невозможно одновременно максимизировать обе метрики. Увеличение recall обычно снижает precision и наоборот.

**Визуализация:**

```
High Threshold (0.8):    Precision ↑↑  Recall ↓↓  (мало тревог, но надёжные)
Optimal Threshold (0.5): Precision ↑   Recall ↑   (баланс)
Low Threshold (0.2):     Precision ↓↓  Recall ↑↑  (много тревог, много ложных)
```

### Выбор метрики для оптимизации

- **Precision важнее**: используй F0.5 или просто отслеживай precision
- **Recall важнее**: используй **F2** ← **наш случай для дождя**
- **Баланс**: используй F1
