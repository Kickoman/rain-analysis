# Baseline Rain Prediction Model (v0.1)

## Обзор

Текущая модель предсказания дождя (`sensor.rain_probability`) работает на основе анализа **dew point spread** (разницы между температурой воздуха и точкой росы) и её **динамики во времени**. Модель реализована как template sensor в Home Assistant и выдаёт вероятность дождя в процентах (0-100%).

**Дата анализа:** 2026-07-12  
**Версия модели:** Production (model_ha_live)  
**Источник кода:** `rainlib.py`, функция `model_ha_live()`

---

## Физический принцип

Модель основана на следующем наблюдении:

> **Когда воздух приближается к насыщению влагой (точка росы близка к температуре), вероятность дождя возрастает.**

### Основные метрики

1. **Dew Point Spread (температурный спред)**
   ```
   spread = temperature - dew_point
   ```
   - Малый spread (< 3°C) → воздух близок к насыщению → высокая вероятность дождя
   - Большой spread (> 10°C) → воздух сухой → низкая вероятность дождя

2. **Spread Derivative (тренд)**
   ```
   spread_deriv = d(spread)/dt  (°C/час)
   ```
   - Отрицательная производная → spread уменьшается → воздух увлажняется → дождь приближается
   - Положительная производная → spread растёт → воздух высыхает → дождь маловероятен

## Алгоритм модели

Модель состоит из трёх компонентов:

### 1. Proximity Score (близость к насыщению)

```python
proximity = clamp(100 - (spread / proximity_divisor * 100), 0, 100)
```

**Параметры:**
- `proximity_divisor = 8.0` — spread при котором proximity = 0%

**Логика:**
- `spread = 0°C` → `proximity = 100%`
- `spread = 8°C` → `proximity = 0%`
- Линейная интерполяция между ними

### 2. Trend Score (динамика изменения)

```python
spread_deriv_3h = derivative(spread, window='3h')
trend_score = clamp(-spread_deriv * trend_gain, trend_floor, trend_ceiling)
```

**Параметры:**
- `trend_gain = 26.7` — множитель для производной
- `trend_floor = -40.0` — максимальное вычитание за негативный тренд
- `trend_ceiling = 40.0` — максимальное добавление за позитивный тренд
- `derivative_window = "3h"` — окно для сглаживания производной

**Логика:**
- Быстрое сужение spread (-1°C/час) → +20 баллов к вероятности
- Быстрое расширение spread (+1°C/час) → -20 баллов

### 3. Взвешенная комбинация (без гистерезиса)

```python
rain_probability = clamp(proximity * 0.7 + trend_score * 0.7, 0, 100)
```

**Параметры:**
- `proximity_weight = 0.7`
- `trend_weight = 0.7`

**Логика:**
- Production модель — stateless (нет гистерезиса)
- Каждый выход зависит только от текущих входных данных
- Для экспериментальной версии с гистерезисом см. `model_tuned` в MODELS.md

## Источники данных

### Локальные датчики
- **Температура:** `sensor.datchik_klimata_temperatura` (балконный датчик)
- **Влажность:** `sensor.datchik_klimata_vlazhnost` (балконный датчик)
- **Давление:** `sensor.office_weather_station_pressure` (DIY метеостанция)
  - ⚠️ **Примечание:** Давление пока **не используется** в baseline модели

### Внешние источники
- **Yandex Weather:** прогноз погоды (для сравнения)
- **Open-Meteo:** ground truth для бэктестинга (осадки)

## Плюсы модели

✅ **Простота и понятность**
- Физически обоснованная модель (dew point spread)
- Легко интерпретируемые параметры
- Не требует машинного обучения

✅ **Работает без давления**
- Можно использовать только с температурой и влажностью
- Подходит для минимальной конфигурации датчиков

✅ **Stateless**
- Простая, предсказуемая модель без скрытого состояния
- Легко отлаживать и тестировать

✅ **Учёт тренда**
- 3-часовое окно сглаживает шум
- Ловит **приближение** дождя, а не только текущее состояние

---

## Минусы модели

❌ **Не использует давление**
- Атмосферное давление — **ключевой предиктор** дождя
- Падение давления часто предшествует дождю за несколько часов
- `sensor.pressure_rain_score` существует но **не интегрирован** в модель

❌ **Ложные срабатывания в сухие ночи**
- Спокойные ночи без ветра → температура падает → spread сужается
- Модель видит это как "приближение дождя"
- Частично решается через `dry_ceiling`, но недостаточно

❌ **Провал во время дождя**
- Сырость во время дождя иногда **снижает** локальную влажность (парадокс)
- Гистерезис помогает, но не полностью решает проблему

❌ **Статические пороги**
- Параметры (`proximity_divisor = 7.0`, etc.) подобраны вручную
- Не адаптируются к сезонным изменениям
- Не учитывают региональную специфику

❌ **Нет учёта внешних факторов**
- Ветер (может разогнать облака или наоборот принести дождь)
- Время суток
- Сезонность
- История за последние дни

## Производительность (известные проблемы)

По данным из README репозитория:

> **"Dry-night false positive"**  
> Обычные спокойные ночи сужают spread так же сильно, как реальные дождливые ночи → одна только пове��хностная влажность не может их разделить. **Это не проблема тюнинга, это отсутствие давления как фичи.**

## Рекомендации по улучшению

### Приоритет 1: Интеграция давления

**Текущее состояние:**
- Давление собирается (`sensor.office_weather_station_pressure`)
- Тренд давления вычисляется (`sensor.diy_weather_station_pressure_trend`)
- Есть начальный `sensor.pressure_rain_score` (сейчас = 0)

**Что нужно:**
- Добавить компонент модели на основе **скорости падения давления**
- Типичный паттерн: падение > 0.5 hPa/час → высокая вероятность дождя
- Интегрировать `pressure_rain_score` в финальную формулу

### Приоритет 2: Адаптивные пороги

- Собрать статистику spread для "дождливых" vs "сухих" ночей
- Сделать `proximity_divisor` зависимым от времени суток
- Добавить сезонную калибровку

### Приоритет 3: Больше контекста

- История за последние 24-48 часов
- Скорость изменения влажности (не только spread, но и абсолютная влажность)
- Направление и скорость ветра (если датчики появятся)

### Приоритет 4: Машинное обучение

- Обучить модель на исторических данных (есть бэктестинг инфраструктура)
- Логистическая регрессия или градиентный бустинг
- Feature engineering: производные второго порядка, rolling windows, lag features

## Заключение

Baseline модель v0.1 — это **хорошая отправная точка**, основанная на физике и здравом смысле. Она работает удивительно хорошо для модели без давления, но имеет известные ограничения.

**Следующий логический шаг:** добавить давление как ключевую фичу. Это устранит основную слабость модели и резко улучшит точность предсказания дождя.
